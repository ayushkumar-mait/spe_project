§ 01

Project overview and topology

This project is a cloud-native DevOps and reliability engineering platform built around a delivery order processing application. The user-facing part is simple: a customer places a delivery order. The important engineering part is everything around that order: queuing, asynchronous processing, health checks, metrics, CI/CD automation, Kubernetes deployment, observability, chaos experiments, and automatic recovery.

The repository is divided by operational responsibility. That makes the system easy to explain in a project demo: every folder answers a specific production engineering question.

services/order-api/ # Spring Boot REST API, dashboard, Swagger, order retry endpoint services/worker/ # Python queue consumer that simulates delivery processing services/healing-controller/ # Python controller that scales or restarts workers libs/platform-common/ # Shared models, Redis repository, queue helpers, structured logging k8s/base/ # Namespace, deployments, services, RBAC, HPA, PDB, topic init job k8s/overlays/ # Local, dev, and prod image overlays chaos/experiments/ # LitmusChaos pod delete, CPU stress, and network latency experiments observability/ # ELK, Filebeat, Logstash, and Kibana setup vault/ # Vault policy and bootstrap script for runtime secrets ansible/ # Role-based deployment automation tools/load-generator/ # Load generator for backlog, HPA, and healing demos Jenkinsfile # Pipeline-as-code for test, build, push, deploy, and rollout verify
Layman explanation
Think of it like a food delivery company. The website accepts orders, a queue holds the orders, delivery staff process them, a dashboard shows status, and an operations system automatically adds more staff or restarts broken workers when the queue becomes unhealthy.
§ 02

Runtime order flow

The application is event-driven. The API does not process the full delivery order inside the HTTP request. It stores the order, publishes a message, and lets workers do the background work. This is what makes the design scalable and resilient.

Customer / Dashboard -> POST /orders -> Spring Boot Order API -> save queued state -> Redis
Order API -> publish delivery_order -> Redpanda topic: jobs -> Worker pods
Workers -> running / completed / failed -> Redis metrics -> Dashboard + Healing Controller
StepComponentWhat happens
1Order APICreates order ID and trace ID, validates JSON, stores queued state.
2RedpandaBuffers the order as a Kafka-compatible message on topic jobs.
3WorkerConsumes messages and simulates payment, restaurant confirmation, rider assignment, route calculation, and notification.
4RedisStores status, result, error, retry metadata, and metrics counters.
5Healing ControllerReads backlog and failure counts, then scales or restarts workers through the Kubernetes API.

§ 04

Application layer

The application is deliberately domain-based, not abstract. Instead of generic jobs, it uses delivery orders, restaurants, addresses, riders, and failed-order retry. This makes the demo easier to explain to classmates while still proving serious DevOps concepts underneath.

4.1 - Spring Boot API endpoints

services/order-api/.../OrderController.javaREST surface
@GetMapping("/healthz")       // liveness: process is alive
@GetMapping("/readyz")        // readiness: Redis dependency reachable
@PostMapping("/orders")       // submit a delivery order
@GetMapping("/orders/{orderId}")
@PostMapping("/orders/{orderId}/retry")
@GetMapping("/metrics")      // queued, running, completed, failed, recovered
API
The controller exposes user-facing operations: submit orders, list recent orders, retry failed orders, and read metrics.
Health
/healthz tells Kubernetes the process is alive. /readyz checks Redis so traffic is only sent to ready pods.
Retry
/orders/{orderId}/retry is the business recovery endpoint: a failed customer order can be submitted again as a clean recovery job.

4.2 - Order submission and retry logic

OrderService.javaqueue producer
public Map<String, Object> placeOrder(PlaceOrderRequest request) {
  Map<String, Object> job = orderJobFactory.createDeliveryJob(request);
  return enqueueJob(job, null);
}

public Map<String, Object> retryFailedOrder(String failedOrderId) {
  Map<String, Object> existing = orderRepository.find(failedOrderId);
  if (!"failed".equals(existing.get("status"))) throw retryNotAllowed;
  Map<String, Object> retryJob = orderJobFactory.createRetryDeliveryJob(existing);
  return enqueueJob(retryJob, failedOrderId);
}
Lines 1-3
A normal order becomes a delivery_order job, then the API stores it and publishes it to Redpanda.
Lines 6-10
A retry is allowed only when the original order is actually failed. The new job uses force_fail=false and keeps a link back to the failed order.

§ 05

The Jenkinsfile - pipeline as code

The Jenkinsfile is the automation heart of the project. It turns a GitHub push into a complete delivery workflow: checkout, tests, image build, Docker Hub push, Kubernetes deploy, and rollout verification. It also includes SCM polling so the project still auto-builds even if an ngrok webhook URL changes.

5.1 - Trigger and parameters

Jenkinsfiletrigger + config
triggers {
  githubPush()
  pollSCM('H/2 * * * *')
}

parameters {
  string(name: 'DOCKERHUB_ORG', defaultValue: 'ayush81080')
  string(name: 'IMAGE_TAG', defaultValue: 'auto')
  booleanParam(name: 'PUSH_IMAGES', defaultValue: true)
  booleanParam(name: 'DEPLOY_TO_K8S', defaultValue: true)
}
Trigger
githubPush() handles GitHub webhook events. pollSCM('H/2 * * * *') checks GitHub every few minutes as a backup.
Images
IMAGE_TAG=auto resolves to the Git commit SHA, which makes every deployed image traceable to source code.
Deploy
The default booleans are true, so a normal push runs the complete build-push-deploy path.

5.2 - Build and push stages

Jenkinsfileimage stages
sh './scripts/run-python-tests.sh'
sh 'mvn -q -f services/order-api/pom.xml test'
sh 'docker build -f services/order-api/Dockerfile -t "$ORDER_API_IMAGE" .'
sh 'docker build -f services/worker/Dockerfile -t "$WORKER_IMAGE" .'

sh 'docker push "$ORDER_API_IMAGE"'

Tests gate the build. Only after Python and Spring Boot tests pass does Jenkins create and push the three service images.

5.3 - Kubernetes deploy stage

Jenkinsfilerollout verification
sh 'kubectl -n "$K8S_NAMESPACE" delete job redpanda-topic-init --ignore-not-found=true || true'
sh 'kubectl apply -k k8s/base'
sh 'kubectl -n "$K8S_NAMESPACE" wait --for=condition=complete job/redpanda-topic-init'
sh 'kubectl -n "$K8S_NAMESPACE" set image deployment/order-api order-api="$ORDER_API_IMAGE"'
sh 'kubectl -n "$K8S_NAMESPACE" set image deployment/job-worker worker="$WORKER_IMAGE"'
sh 'kubectl -n "$K8S_NAMESPACE" set image deployment/healing-controller healing-controller="$HEALER_IMAGE"'
sh 'kubectl -n "$K8S_NAMESPACE" rollout status deployment/order-api --timeout=240s'
Important Jenkins fix
Kubernetes Job pod templates are immutable. The pipeline deletes the old redpanda-topic-init job before applying manifests so repeated deployments do not fail with a "field is immutable" error.

§ 07

Kubernetes manifests

Kubernetes runs the system as a cluster application. The important resources are Deployments for long-running services, Services for stable networking, HPA for scaling, RBAC for the healing controller, PDBs for availability, and a Job to initialize the Redpanda topic.

7.1 - Order API deployment

k8s/base/order-api.yamlzero-downtime API rollout
replicas: 2
strategy:
  rollingUpdate:
    maxUnavailable: 0
    maxSurge: 1
readinessProbe:
  httpGet: { path: /readyz, port: http }
livenessProbe:
  httpGet: { path: /healthz, port: http }
Replicas
Two API pods keep the service available during updates.
Rolling
maxUnavailable: 0 means Kubernetes should not intentionally take the API below available capacity during rollout.
Probes
Readiness protects traffic. Liveness protects recovery by restarting unhealthy containers.

7.2 - Worker and HPA

k8s/base/hpa.yamlautoscaling
kind: HorizontalPodAutoscaler
metadata:
  name: job-worker-hpa
spec:
  minReplicas: 1
  maxReplicas: 8
  metrics:
    - resource:
        name: cpu
        target: { type: Utilization, averageUtilization: 60 }

HPA reacts to CPU pressure. The custom healing controller reacts to application pressure such as queue backlog and failed orders. Together, they cover infrastructure load and business-level symptoms.

7.3 - Redpanda topic init job

k8s/base/redpanda-topic-job.yamlqueue bootstrap
brokers="redpanda.job-platform.svc.cluster.local:9092"
until rpk cluster info -X brokers="$brokers" >/dev/null 2>&1; do sleep 1; done
partitions="$(rpk topic describe jobs ... | awk '/PARTITIONS/ {print $2}' || true)"
if [ -z "$partitions" ]; then rpk topic create jobs -p 6 -r 1; fi

The jobs topic is created with six partitions so multiple worker replicas can process messages concurrently. Without multiple partitions, scaling workers would look good on paper but would not produce meaningful parallel queue consumption.

Ansible roles

Ansible gives the project a modular deployment automation layer. Jenkins is the CI/CD engine, but Ansible describes deployment responsibilities as reusable roles.

ansible/site.yml # calls all roles in order ansible/roles/k8s-role/tasks/ # applies base Kubernetes infrastructure ansible/roles/app-role/tasks/ # sets app images and waits for rollouts ansible/roles/monitoring-role/tasks/ # applies ELK and Filebeat manifests ansible/roles/vault-role/tasks/ # verifies Vault policy files and prints setup command

  gather_facts: false

In a viva, this is how to explain it: Ansible is used for repeatable operations. Instead of manually typing every deployment command, roles organize the cluster setup, application rollout, monitoring setup, and Vault verification.

§ 09

ELK observability

The application emits structured JSON logs. ELK collects those logs and makes them searchable. In Docker Compose, services use the GELF logging driver to Logstash. In Kubernetes, Filebeat collects container logs and forwards them to Logstash.

9.1 - Logstash pipeline

observability/logstash/pipeline/logstash.confstructured log parsing
input {
  gelf { port => 12201 }
  tcp  { port => 5000 codec => json_lines }
}
filter {
  if [message] =~ "^{.*}$" { json { source => "message" target => "app" } }
  mutate { add_field => { "project" => "automated-chaos-testing-self-healing-microservices-platform" } }
}
output { elasticsearch { index => "job-platform-logs-%{+YYYY.MM.dd}" } }
EventWhere it comes fromWhy it matters
order_submittedOrder APIProves API accepted and queued a delivery order.
job_processing_completedWorkerProves a worker consumed and completed the job.
order_retry_submittedOrder APIProves failed business work was resubmitted.
healing_action_selectedHealing controllerProves the controller made a scale or restart decision.
Local laptop note
ELK is memory-heavy. If Docker Desktop has less than about 6 GB memory available, the core platform can run fine while Kibana/Logstash may become unstable. For a safe demo, run ELK only after increasing Docker memory or use the prepared report screenshots.

§ 11

Chaos and self-healing

Chaos engineering intentionally breaks part of the system to prove that recovery works. This repo includes LitmusChaos experiments for pod deletion, CPU stress, and network latency. It also supports manual chaos with a simple worker pod delete command.

11.1 - Healing policy

services/healing-controller/src/healing_controller/policy.pydecision rules
if snapshot.backlog > config.queued_threshold:
    actions.append(HealingAction(action="scale", target_replicas=target))

if snapshot.failed >= config.failed_threshold:
    actions.append(HealingAction(action="restart"))

if snapshot.backlog == 0 and snapshot.current_replicas > config.min_replicas:
    actions.append(HealingAction(action="scale", target_replicas=config.min_replicas))
ProblemSignalRecovery action
High queue pressureQueued + running orders above thresholdScale worker deployment upward
Repeated failuresFailed orders reach thresholdRestart worker deployment
Backlog goneQueued + running returns to zeroScale workers back to minimum
Worker pod deletedDeployment has fewer pods than desiredKubernetes recreates replacement pod

§ 13

Putting it all together

This project is not just a delivery order app. It is a complete DevOps reliability platform: the app creates real work, Jenkins automates delivery, Docker packages services, Kubernetes orchestrates them, Redpanda and Redis provide queue/state infrastructure, ELK observes behavior, Vault demonstrates secret handling, Ansible organizes operations, and chaos experiments prove recovery.

"Our project is a delivery microservices platform. Customers submit orders through a Spring Boot API. The API stores order state in Redis and publishes work to Redpanda. Python workers consume the queue and update order status. Jenkins automatically tests, builds, pushes Docker images, and deploys to Kubernetes. Kubernetes handles pod recovery and autoscaling, while our custom healing controller reacts to application-level backlog and failures.
